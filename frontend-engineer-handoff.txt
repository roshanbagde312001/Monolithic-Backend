APP DEFEND FRONTEND HANDOFF
Version: 1
Audience: Frontend engineer building the primary web application UI

1. PRODUCT SUMMARY

Build a modern React.js frontend for App Defend, an enterprise application security platform.
The frontend should feel premium, clean, modern, and operationally serious.
This is not a generic admin template. Avoid flat boilerplate dashboards.

Core modules currently available in backend:
- Authentication with JWT access token and refresh token
- User management
- RBAC management for roles, permissions, and views
- OEM integrations
- GitLab integration sync for namespaces, groups, and projects
- License install, license status, audit, and capacity reporting
- Health endpoint

2. REQUIRED FRONTEND STACK

Use this stack unless there is a strong implementation reason not to:
- React 18+
- TypeScript
- Vite
- React Router
- TanStack Query for server state
- Zustand or Context API for auth/session state
- Axios for API client
- React Hook Form for forms
- Zod for schema validation
- Tailwind CSS or CSS Modules
- Heroicons or Lucide React for icons

Preferred supporting patterns:
- Feature-based folder structure
- Typed API client wrappers
- Protected routes
- Reusable layout shell
- Centralized error handling
- Token refresh interceptor

3. VISUAL DIRECTION

The application should look modern, controlled, and enterprise-ready.
Design intent:
- light theme first
- clean neutral base with one strong brand accent
- spacious layout
- strong typography hierarchy
- subtle gradients or texture, not loud gamer-style visuals
- polished cards, tables, filters, and empty states
- smooth hover and loading states

Avoid:
- default template look
- purple-heavy generic dashboards
- overly rounded toy-like interfaces
- dense legacy admin layouts

Suggested look and feel:
- top navigation plus contextual page header
- left navigation for core modules on authenticated screens
- searchable tables
- summary cards on home/dashboard
- activity and status blocks
- account menu in top-right

4.3 Sidebar navigation

Minimum navigation items:
- Home
- Users
- RBAC
- Integrations
- GitLab
- Licenses
- Audit / Reports

Navigation must be filtered by access control data from the authenticated user object.

Role / permission / view behavior required:
- use user.roles, user.permissions, and user.views from login response
- page visibility should be controlled at navigation level
- route access should be controlled at route level
- action buttons should be controlled at component level

Access-control rules for frontend:
- roles are high-level user grouping labels
- permissions control whether actions are visible or executable
- views control whether user should see a page or section in navigation

Required frontend utilities:
- hasRole(roleCode)
- hasPermission(permissionCode)
- hasView(viewCode)
- canAccessPage(viewCode, fallbackPermissions)
- guardRoute(viewCode or permissionCode)

Minimum view mapping for initial pages:
- Home page -> DASHBOARD
- Users page -> USER_MANAGEMENT
- RBAC page -> RBAC_MANAGEMENT
- Integrations page -> INTEGRATION_MANAGEMENT
- Licenses page -> LICENSE_MANAGEMENT

Minimum permission mapping for actions:
- list users -> USER_READ
- create/update/delete users -> USER_WRITE
- list roles -> ROLE_READ
- create/update/delete roles -> ROLE_WRITE
- assign permissions/views to role -> ROLE_WRITE
- list permissions -> PERMISSION_READ
- create/update/delete permissions -> PERMISSION_WRITE
- list views -> VIEW_READ
- create/update/delete views -> VIEW_WRITE
- integrations list/test -> INTEGRATION_READ
- integrations create/update/delete -> INTEGRATION_WRITE
- license status -> LICENSE_READ
- license install/encrypted install/create bundle -> LICENSE_WRITE
- license audit -> LICENSE_AUDIT_READ
- license reports -> LICENSE_REPORT_READ

UI behavior requirement:
- hide inaccessible pages from sidebar
- block direct route access to unauthorized pages
- hide unauthorized action buttons
- show read-only mode where sensible
- show a clean unauthorized state rather than crashing

6. AUTH FLOW

6.1 Login

Endpoint:
- POST /api/v1/auth/login

Request:
{
  "email": "admin@appdefend.local",
  "password": "Admin@123"
}

Response fields:
- accessToken
- refreshToken
- expiresInSeconds
- user

Login response shape:
{
  "accessToken": "<jwt>",
  "refreshToken": "<jwt>",
  "expiresInSeconds": 1800,
  "user": {
    "id": 1,
    "fullName": "Platform Admin",
    "email": "admin@appdefend.local",
    "roles": ["SUPER_ADMIN"],
    "permissions": ["USER_READ", "USER_WRITE", "ROLE_READ"],
    "views": ["DASHBOARD", "USER_MANAGEMENT", "RBAC_MANAGEMENT"]
  }
}

6.2 Refresh token

Endpoint:
- POST /api/v1/auth/refresh

Request:
{
  "refreshToken": "<refresh-token>"
}

6.3 Logout

Endpoint:
- POST /api/v1/auth/logout

Request:
{
  "accessToken": "<access-token>",
  "refreshToken": "<refresh-token>"
}

6.4 Frontend auth handling requirements

- store tokens securely in memory plus persistent storage if needed
- add Authorization header: Bearer <accessToken>
- on 401, try refresh once
- if refresh fails, clear session and redirect to login
- store logged-in user object from login response
- derive role/permission/view based UI state from login response

Create / update integration payload:
{
  "name": "GitLab SaaS Primary",
  "providerType": "GITLAB",
  "deploymentMode": "SAAS",
  "namespacePath": "my-group/platform",
  "baseUrl": "https://gitlab.com/api/v4",
  "credentialsJson": "{\"token\":\"<gitlab-token>\"}",

SaaS GitLab URL rule:
- `baseUrl` is the GitLab host/API root used by the backend, for example `https://gitlab.com/api/v4`
- `namespacePath` is the path segment under that host, for example `my-group/platform`
- frontend should visually show the composed namespace URL as `https://gitlab.com/my-group/platform`

For self-managed GitLab:
{
  "name": "GitLab Self Managed",
  "providerType": "GITLAB",
  "deploymentMode": "SELF_MANAGED",

8. GITLAB FRONTEND UX RULES

When user selects providerType = GITLAB:
- show deployment mode selector
- allowed values:
  - SAAS
  - SELF_MANAGED

If deployment mode = SAAS:
- show namespace text input
- user manually enters namespacePath such as `my-group/platform`
- show a live preview of the effective namespace URL using the GitLab host plus namespacePath
- on save, persist selected namespacePath
- on sync, backend must fetch groups and projects only for that namespace

If deployment mode = SELF_MANAGED:
- hide namespace input
- sync all accessible groups and projects

GitLab groups and projects should be rendered in separate tables.
Each row should have:
- human-readable columns
- raw JSON view action
- copy path/full-path action where useful

14. SECURITY AND SESSION UX

Required:
- logout action in account dropdown
- session expiry handling
- unauthorized page when 403 is returned
- redirect to login on session loss
- permission-aware hiding or disabling of page-level and action-level UI
- route guards based on view and permission checks

Frontend access-control implementation notes:
- after login, cache user.roles, user.permissions, and user.views
- build a centralized access-control hook or utility module
- do not hardcode visibility logic inside random components
- use declarative wrappers such as:
  - ProtectedRoute
  - RequirePermission
  - RequireView
  - PageAccessBoundary
